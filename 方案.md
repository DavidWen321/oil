# 流式响应优化方案 — 诊断报告与改进计划

## 一、当前项目诊断

### 1.1 后端流式粒度：Token 级 ✅ 已达标

后端实现是正确的。`workflows/graph.py:324` 使用 LangGraph 的 `astream_events(version="v2")`，
在 `on_chat_model_stream` 事件中逐 token yield：

```python
# graph.py:329-334
if kind == "on_chat_model_stream":
    chunk = event["data"].get("chunk")
    if chunk and hasattr(chunk, "content") and chunk.content:
        if not (hasattr(chunk, "tool_calls") and chunk.tool_calls):
            final_content += chunk.content
            yield {"type": "token", "content": chunk.content}  # 每个token立即推送
```

`StreamEventProcessor` 将每个 token 映射为 `CONTENT_DELTA` SSE 事件，无缓冲直接 yield。
SSE 端点 `chat_v2.py:50` 也是逐包 yield，没有节点级阻塞。

**结论：后端不是卡顿瓶颈。**

### 1.2 前端 rAF 缓冲：已实现 ✅ 但有隐患

`useAgentTrace.ts:509-519` 对 `content_delta` / `response_chunk` 事件做了 requestAnimationFrame 批量刷新：

```typescript
case 'content_delta':
case 'response_chunk': {
  chunkBufferRef.current += chunk;
  if (rafIdRef.current === null) {
    rafIdRef.current = requestAnimationFrame(flushBuffer); // ~16ms 批量
  }
  return prev; // 不立即触发 setState
}
```

这意味着每 16ms 刷新一次，多个 token 会合并为一次 React 渲染。这是正确的做法。

**但问题在于：** 这只是"防爆发"，不是"匀速平滑"。如果后端 Agent 思考 3 秒后突然吐出 200 个 token，
rAF 会在 ~200ms 内全部渲染完（约 12 帧），视觉上仍然是"卡顿→爆发"。

### 1.3 闪烁根因：组件切换 ❌ 核心问题

`AIChat.tsx:126-140` 存在两套渲染路径：

```tsx
{/* 路径 A：历史消息（非流式） */}
{messages.map((msg, index) => (
  <ChatMessage key={`${msg.role}-${index}`} role={msg.role} content={msg.content} />
))}

{/* 路径 B：流式消息（独立实例） */}
{sending && (
  <ChatMessage role="assistant" content={finalResponse} streaming={true} tools={activeTools} />
)}
```

当 `status` 变为 `completed` 时：
1. `sending` 变 false → 路径 B 的 ChatMessage **卸载**（cursor 消失、StreamingMarkdown 销毁）
2. `useEffect` 将 finalResponse 写入 messages 数组 → 路径 A **挂载**新的 ChatMessage
3. 两帧之间出现视觉断层 = **闪烁**

### 1.4 StreamingMarkdown 的二次闪烁

`StreamingMarkdown.tsx:85-96` 在 streaming 和非 streaming 之间切换了完全不同的 DOM 结构：

```tsx
// streaming=false 时：单个 MarkdownRenderer
if (!streaming || !content) {
  return <MarkdownRenderer content={content} />;  // 结构 A
}
// streaming=true 时：div 容器 + 分块渲染
return (
  <div className={styles.streamingContainer}>   // 结构 B
    {memoizedCompleted}
    <div className={styles.activeBlock}>
      <MarkdownRenderer content={fixedActiveBlock} />
    </div>
  </div>
);
```

流式结束时 DOM 从结构 B 切换到结构 A，React 会销毁整棵子树并重建，这是第二层闪烁来源。

### 1.5 诊断总结

| 层级 | 状态 | 问题 |
|------|------|------|
| 后端 SSE 粒度 | ✅ Token 级 | 无 |
| rAF 批量刷新 | ✅ 已实现 | 缺少匀速平滑队列 |
| 组件切换闪烁 | ❌ 严重 | 流式/完成态是两个不同的组件实例 |
| Markdown 结构切换 | ❌ 中等 | streaming/非streaming 返回不同 DOM 树 |
| 光标实现 | ✅ 已有 | 随组件卸载消失，应独立于组件生命周期 |
| 代码块容错 | ✅ 已有 | fixUnclosedFences 处理了未闭合的 ``` |

## 二、业界标杆对比

### 2.1 ChatGPT (GPT-5.3)

| 技术点 | 实现方式 |
|--------|---------|
| 传输协议 | SSE，`text/event-stream`，增量 JSON delta |
| 流式粒度 | Token 级（每 2-4 字符一个 delta） |
| Markdown 渲染 | 增量渲染，未闭合标签乐观补全 |
| 组件切换 | **无切换**，始终同一个组件实例，完成时仅隐藏光标 |
| 平滑队列 | 有，前端维护 charQueue + 固定节奏消费（~20ms/字符） |
| 光标 | `▋` 伪元素，CSS animation，流结束移除 class 而非卸载组件 |
| Artifact | 独立 iframe 渲染，不影响主消息流 |

### 2.2 Claude (Opus 4.6 Web / Claude Code CLI)

| 技术点 | 实现方式 |
|--------|---------|
| 传输协议 | SSE，多通道事件（content_block_delta / thinking / tool_use） |
| 流式粒度 | Token 级 + Extended Thinking 流 |
| Markdown 渲染 | 增量渲染，代码块实时高亮 |
| 组件切换 | **无切换**，消息组件从创建到完成是同一个 React 实例 |
| 平滑队列 | 有，内部 buffer + rAF 节流 |
| CLI 模式 | Claude Code 全程流式，`stream=true`，逐字输出到终端 |
| 工具调用 | 流式通道内联 tool_use event，不中断文本流 |

### 2.3 Gemini (3 Pro)

| 技术点 | 实现方式 |
|--------|---------|
| 传输协议 | SSE / WebSocket（Live API 用 WebSocket） |
| 流式粒度 | Token 级，Gemini Flash 极快需要前端限速 |
| Markdown 渲染 | 增量渲染 + 代码块实时折叠 |
| 组件切换 | **无切换** |
| 平滑队列 | 有，且针对 Flash 模型做了更激进的限速（~32ms/批） |

### 2.4 核心共识

三大平台的共同做法：
1. **绝不切换组件** — 流式和完成态是同一个组件实例，完成时只改 CSS class
2. **匀速消费队列** — 前端不直接渲染收到的 token，而是放入队列按固定节奏消费
3. **光标是 CSS 伪元素** — 不是独立 DOM 节点，通过 class 控制显隐

## 三、改进方案（4 步）

### 步骤 1：消除组件切换闪烁（AIChat.tsx）

**原理：** 不再条件渲染两个 ChatMessage，而是始终在 messages 数组末尾维护一个"活跃消息"。

**改动文件：** `src/pages/ai/AIChat.tsx`

```tsx
// 改前：两套渲染路径
{messages.map((msg, index) => (
  <ChatMessage key={...} role={msg.role} content={msg.content} />
))}
{sending && (
  <ChatMessage role="assistant" content={finalResponse} streaming={true} tools={activeTools} />
)}

// 改后：统一渲染路径
const displayMessages = useMemo(() => {
  const base = [...messages];
  if (sending && finalResponse !== undefined) {
    // 流式进行中：末尾追加/更新活跃消息
    if (base.length > 0 && base[base.length - 1].role === 'assistant' && base[base.length - 1].streaming) {
      base[base.length - 1] = { role: 'assistant', content: finalResponse, streaming: true, tools: activeTools };
    } else {
      base.push({ role: 'assistant', content: finalResponse, streaming: true, tools: activeTools });
    }
  }
  return base;
}, [messages, sending, finalResponse, activeTools]);

// JSX 中只有一套渲染：
{displayMessages.map((msg, index) => (
  <ChatMessage
    key={`msg-${index}`}
    role={msg.role}
    content={msg.content}
    streaming={msg.streaming}
    tools={msg.tools}
  />
))}
```

**效果：** 流式消息和完成消息是同一个 React 实例（相同 key），不会卸载重建。
完成时只是 `streaming` 从 true 变 false，光标消失，DOM 结构不变。

### 步骤 2：统一 StreamingMarkdown 的 DOM 结构

**原理：** 无论 streaming 是否为 true，始终返回相同的 DOM 结构。

**改动文件：** `src/components/agent/StreamingMarkdown.tsx`

```tsx
// 改前：两种不同的 DOM 结构
if (!streaming || !content) {
  return <MarkdownRenderer content={content} />;        // 结构 A
}
return <div className={styles.streamingContainer}>...</div>; // 结构 B

// 改后：始终使用分块结构
export default function StreamingMarkdown({ content, streaming }: StreamingMarkdownProps) {
  const blocks = useMemo(() => splitBlocksPreservingFences(content || ''), [content]);
  const completedBlocks = blocks.slice(0, -1);
  const activeBlock = blocks[blocks.length - 1] || '';

  const memoizedCompleted = useMemo(
    () => completedBlocks.map((block, i) => (
      <div key={`block-${i}`} className={styles.block}>
        <MarkdownRenderer content={block} />
      </div>
    )),
    [completedBlocks],
  );

  const fixedActiveBlock = streaming ? fixUnclosedFences(activeBlock) : activeBlock;

  // 始终返回相同结构，streaming 只影响是否修复未闭合标签
  return (
    <div className={styles.streamingContainer}>
      {memoizedCompleted}
      <div className={styles.activeBlock}>
        <MarkdownRenderer content={fixedActiveBlock} />
      </div>
    </div>
  );
}
```

**效果：** 流式结束时 DOM 树不重建，只是最后一个 block 的内容从"修复版"变为"原始版"（通常一样）。

### 步骤 4：匀速平滑队列（进阶优化）

**原理：** 当前 rAF 缓冲只解决了"防止每个 token 触发一次渲染"的问题，
但没有解决"Agent 思考 3 秒后突然吐出大量 token"的视觉爆发感。

需要在 rAF 和 setState 之间加一层"匀速消费队列"。

**改动文件：** `src/hooks/useAgentTrace.ts`（修改 flushBuffer 逻辑）

```typescript
// 新增：匀速消费配置
const CHARS_PER_FRAME = 8;  // 每帧最多消费 8 个字符（~128 字符/秒 @ 60fps）
const smoothQueueRef = useRef('');

const flushBuffer = useCallback(() => {
  rafIdRef.current = null;

  // 将新到达的 chunk 追加到平滑队列
  smoothQueueRef.current += chunkBufferRef.current;
  chunkBufferRef.current = '';

  if (!smoothQueueRef.current) return;

  // 每帧只消费固定数量的字符
  const toConsume = smoothQueueRef.current.slice(0, CHARS_PER_FRAME);
  smoothQueueRef.current = smoothQueueRef.current.slice(CHARS_PER_FRAME);

  setState(prev => ({
    ...prev,
    finalResponse: prev.finalResponse + toConsume,
  }));

  // 队列未空，继续调度下一帧
  if (smoothQueueRef.current) {
    rafIdRef.current = requestAnimationFrame(flushBuffer);
  }
}, []);
```

**效果：**
- 后端突然吐出 200 个字符 → 前端用 ~25 帧（~400ms）匀速渲染完
- 视觉上是平滑的打字机效果，而非"卡顿→爆发"
- `CHARS_PER_FRAME` 可调：值越大越快但越"爆发"，值越小越平滑但延迟越高
- 推荐值：6-12（对应 ~100-200 字符/秒，接近人类快速阅读速度）

**注意：** `drainBufferedChunks` 也需要同步清空 `smoothQueueRef`：

```typescript
const drainBufferedChunks = useCallback(() => {
  if (rafIdRef.current !== null) {
    cancelAnimationFrame(rafIdRef.current);
    rafIdRef.current = null;
  }
  const buffered = chunkBufferRef.current + smoothQueueRef.current;
  chunkBufferRef.current = '';
  smoothQueueRef.current = '';
  return buffered;
}, []);
```

## 五、优先级建议

| 优先级 | 步骤 | 效果 |
|--------|------|------|
| P0 必做 | 步骤 1：消除组件切换 | 彻底解决闪烁 |
| P0 必做 | 步骤 2：统一 DOM 结构 | 消除二次闪烁 |
| P1 建议 | 步骤 3：伪元素光标 | 更优雅的光标消失过渡 |
| P2 锦上添花 | 步骤 4：匀速队列 | 丝滑打字机效果 |

步骤 1+2 是解决闪烁的核心，必须一起做。步骤 3+4 是体验优化，可以后续迭代。
