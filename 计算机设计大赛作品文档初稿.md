## 需求分析

### 时代背景与项目愿景：推动管道能耗管理数字化与分析科学化

在能源输送与工业运行日益向数字化、精细化、智能化方向发展的背景下，油气管道作为重要的能源输送基础设施，其运行安全、输送效率与能耗水平越来越受到重视。尤其是在长距离输油管道场景中，管道参数、油品物性、泵站工况、运行压力等因素彼此耦合，系统运行过程复杂，仅依靠人工经验进行分析，往往存在效率低、误差大、决策滞后等问题。

在传统工作模式下，管道项目基础数据、设备参数与运行结果通常分散在不同表格、文档或系统中，管理人员需要手工整理项目、管道、泵站与油品等信息，再结合经验完成水力分析与泵站调度判断。这种方式不仅数据组织成本高，而且在面对多项目、多参数、多工况分析时，难以快速得出可靠结论，也难以形成可持续积累与复用的知识体系。

本项目立足于管道能耗分析与运行优化场景，旨在构建一个集“基础数据管理、专业计算分析、节能优化辅助、知识智能问答、动态报告生成”于一体的综合平台。系统通过整合管道运行相关数据，构建规范化的数据组织方式；通过水力分析、泵站优化与敏感性分析提升专业分析能力；通过知识库检索与智能体技术增强自然语言交互与知识辅助能力，从而推动管道能耗管理由传统经验式分析向数字化、模型化、智能化方向转变。

### 面向对象与价值定位：服务管道运行管理需求，提升分析与决策效率

本系统面向的核心用户主要包括管道运行管理人员、工程技术人员、设备维护人员、项目负责人以及相关教学与科研场景下的学习使用者。不同用户虽然关注重点不同，但都需要围绕“真实参数获取、专业分析判断、节能优化决策、历史结果沉淀”这一主线开展工作。

对于运行管理人员而言，系统能够帮助其快速掌握项目、管道与泵站的核心参数，直观查看计算结果与运行特征，提高能耗管理效率；对于工程技术人员而言，系统提供可重复调用的专业计算能力，能够辅助完成水力分析、泵站组合优化与参数敏感性评估；对于维护与优化人员而言，系统提供异常分析线索、知识问答支持与报告生成功能，增强问题识别与决策支撑能力；对于教学与科研使用者而言，系统兼具工程实践场景和智能分析特征，能够为专业学习、课程设计与研究探索提供较强的支撑平台。

因此，本项目的价值定位不是单一功能型软件，而是一个面向管道能耗分析场景的综合性智能平台。它既强调专业分析结果的准确性，也强调系统使用过程的便捷性、知识支撑能力和持续扩展能力。

### 核心功能创新：AI赋能管道能耗智能分析

本项目围绕管道能耗分析这一专业场景，在传统业务管理与计算分析基础上，融入人工智能与知识增强能力，形成以下几个方面的核心功能创新：

1. 基础数据与计算分析一体化。系统将项目、管道、泵站、油品等基础数据管理与水力分析、节能优化、敏感性分析进行统一整合，避免传统分析中“数据分散、重复录入、计算脱节”的问题。

2. 面向工程场景的专业计算能力。系统针对油气管道运行场景实现了水力分析策略、泵站优化策略与敏感性分析策略，能够基于实际参数完成雷诺数、流态、摩阻损失、水力坡降、首末站压力以及泵组组合推荐等核心指标计算。

3. AI知识问答与智能辅助分析。系统引入知识库与大模型能力，支持围绕管道规范、计算原理、操作流程、典型案例等内容进行自然语言问答，使平台不仅能够“算数据”，还能够“答问题”“给建议”。

4. 自然语言驱动的智能交互能力。平台通过智能体工作流将用户自然语言请求与数据库查询、知识检索、计算服务调用连接起来，使用户能够以更自然的方式使用复杂功能，降低系统使用门槛。

5. 动态报告与分析结果沉淀。系统支持将计算历史、统计指标与业务数据结合，生成动态分析报告，并对结果进行归档，提升分析过程的可追溯性与复用价值。

### 竞争优势分析

与仅具备单一计算能力的传统分析工具相比，本系统具有“数据管理 + 专业分析 + 智能交互 + 知识支撑 + 报告输出”的完整闭环优势。它不仅能完成单次计算任务，还能围绕项目全流程提供统一平台支撑。

与仅提供基础业务管理的信息系统相比，本系统进一步引入专业计算模型与 AI 智能体能力，使平台具备更强的行业针对性和智能分析深度。系统不仅保存数据，更能够基于数据进行专业分析、趋势判断和优化建议。

与通用型问答机器人或普通大模型应用相比，本系统的智能能力建立在真实业务数据、专业知识库和计算服务之上。AI 不是独立悬浮的聊天功能，而是与数据库查询、后端计算、知识检索和报告生成深度结合，具有更高的业务可用性和工程场景适配能力。

### 技术架构亮点：微服务驱动、精准计算、智能扩展

本项目在技术架构上采用前后端分离与微服务协同的总体设计。前端基于 React + TypeScript + Vite 构建，结合 Ant Design、ECharts 等组件与图表技术，实现仪表盘展示、基础数据管理、参数录入、分析结果可视化和智能对话等功能。系统界面支持项目统计、管道参数散点图、泵站参数折线图、油品参数热力图等多种展示方式，兼顾了业务交互效率与可视化表达能力。

后端采用 Spring Cloud Alibaba 微服务架构，划分为网关服务、认证服务、系统服务、数据服务和计算服务等模块。系统通过网关统一接入请求，通过 Sa-Token 与 Redis 实现登录认证与会话校验，通过 Nacos 实现服务注册与配置管理，通过 MyBatis Plus 和 MySQL 实现业务数据存储与访问，从而保证系统具有较好的模块解耦能力、可维护性和后续扩展能力。

在计算层面，系统围绕管道工程场景实现了专业化分析引擎。计算服务采用 Java 高精度 BigDecimal 进行水力计算，保证工程参数分析过程中数值处理的稳定性与精确性。系统内置水力分析策略、泵站优化策略和敏感性分析策略，能够对不同工况进行结构化计算与结果输出。

在智能层面，平台额外集成基于 Python FastAPI 的智能体服务，结合 LangChain、LangGraph、Milvus 等技术实现知识检索、自然语言问答、工具调用与动态报告生成。该架构使系统在传统信息化平台基础上，进一步具备了知识增强、语义理解和智能交互能力，为后续扩展更复杂的智能分析场景预留了充分空间。

### AI智慧集成功能：打造智能化管道分析助手

为了增强系统在管道分析场景下的智能辅助能力，平台在整体架构中集成了大模型、知识库、向量检索与智能体编排机制，形成面向业务问题的智能辅助体系。系统支持通过 OpenAI 兼容接口接入大语言模型，并结合业务工具调用机制，使 AI 能够围绕项目数据、知识内容和计算服务进行协同工作。

在知识问答方面，系统构建了面向管道工程的知识库，内容包括技术规范、计算公式、操作规程、案例经验和常见问题等。系统通过文档处理、语义分块、向量化存储与混合检索等方式，为专业问答提供可检索、可引用的知识支撑。

在智能分析方面，系统将大模型与工具调用相结合，使智能体能够根据用户自然语言需求自动选择数据库查询、知识检索、计算调用等工具，实现“问题理解—任务拆解—工具执行—结果综合”的智能分析流程。该能力显著提升了平台的人机交互自然度，也增强了复杂业务场景下的辅助决策能力。

## 概要设计

### 系统总体方案设计

本系统按照“业务数据管理层、专业计算分析层、智能辅助服务层、前端展示交互层”进行总体设计。各模块既相互配合，又保持相对独立，形成清晰的系统层次结构。

在业务数据管理层，系统负责项目、管道、泵站、油品等基础对象的统一管理，保证后续分析所需参数来源清晰、结构规范；在专业计算分析层，系统围绕输油管道场景实现水力分析、泵站优化和敏感性分析；在智能辅助服务层，系统接入知识库、智能体工作流和报告生成服务，用于完成自然语言问答、智能辅助分析和报告输出；在前端展示交互层，系统通过仪表盘、表单、图表、对话界面等方式完成用户交互与结果呈现。

这种设计使系统既能够满足传统业务系统对稳定性和可维护性的要求，又能够满足智能化场景对扩展性和灵活性的要求。

### 数据管理模块

数据管理模块主要承担系统基础业务数据的录入、维护与组织任务，是整个分析平台运行的基础。系统围绕管道能耗分析场景设计了项目管理、管道参数管理、泵站参数管理和油品物性管理等功能页面，使用户可以对分析所需的核心对象进行统一维护。

其中，项目信息用于组织业务对象并建立分析归属关系；管道参数用于描述管道长度、外径、壁厚、粗糙度、高程等关键物理参数；泵站参数用于记录泵效率、电机效率、排量、进站压头及不同泵型扬程等信息；油品物性数据则用于记录密度、黏度等与计算直接相关的关键参数。通过这一模块，系统实现了专业分析参数的结构化与集中化管理，为后续计算分析提供标准输入。

### 水力分析模块

水力分析模块是本系统的核心专业功能之一，用于对管道输送工况进行基础分析。用户可以手动输入参数，也可以直接从已有项目、管道、油品和泵站数据中带入分析参数。系统根据输入内容，自动完成雷诺数、流态判断、摩阻损失、水力坡降、总扬程、首站出站压头、末站进站压头等关键指标的计算，并通过图表展示压力变化趋势。

该模块的设计目标是将传统复杂的工程计算过程转化为可视化、结构化、可反复调用的系统功能，使工程分析工作更加标准化、便捷化。

### 泵站优化模块

泵站优化模块用于对不同泵组运行方案进行比较和推荐。系统依据输入的流量、油品参数、管道参数、泵站效率及电价等信息，对不同泵组组合进行遍历计算，并筛选满足运行约束条件的可行方案，从中选择更优的节能运行组合。

系统不仅输出推荐的 ZMI480 与 ZMI375 泵组开启数量，还会同步给出总扬程、总压降、末站进站压头、年能耗、预计总成本和推荐说明等结果。该模块使用户能够更加直观地理解不同调度方案的经济性与可行性，提高节能优化分析效率。

### 敏感性分析模块

敏感性分析模块用于研究关键输入参数变化对分析结果的影响程度。系统支持围绕流量、油品密度、油品黏度、管道直径、管道粗糙度等变量进行单因子分析，并输出敏感性曲线、影响幅度、敏感系数和排名结果。

通过该模块，用户可以识别影响系统运行结果的关键因素，判断不同变量变化对末站压力与摩阻损失的影响程度，从而为参数优化、风险识别和后续决策提供依据。

### 仪表盘与可视化模块

为了增强系统的数据感知能力和展示效果，平台设计了综合仪表盘页面，用于对项目、管道、泵站、油品和历史记录进行可视化展示。系统通过统计卡片、柱状图、散点图、折线图和热力图等方式，对关键数据特征进行集中呈现。

这种可视化设计不仅提升了系统的观感与展示效果，也便于用户从全局视角快速理解系统数据结构与运行状态，为比赛展示与项目答辩提供了良好的表现形式。

### AI对话与知识录入模块

本系统设计了独立的 AI 对话与知识录入模块。知识录入模块支持上传 Markdown、TXT、PDF、DOCX 等格式文档，并按照标准规范写入知识库。AI 对话模块支持围绕一般问答、故障诊断模式和优化建议模式与系统交互，用户可以以自然语言输入业务问题，系统将结合当前工具链与知识库给出回答。

该模块的加入，使系统从传统“菜单式操作”进一步扩展到“自然语言驱动分析”，提升了系统的人机交互体验和业务适应能力。

### 报告与历史沉淀模块

系统通过计算历史、统计接口和动态报告能力，将每次分析结果进行结构化记录，并支持后续查询、分页展示和报告归档。通过结果沉淀，系统可以逐步积累可复用的分析历史和报告素材，增强平台在长期使用过程中的价值。

该模块不仅有助于结果追溯和工作留档，也为后续基于历史数据开展趋势分析、对比分析和智能总结奠定基础。

## 详细设计

### 系统整体架构设计

本系统采用前后端分离与微服务协同架构，整体上由前端应用、Java 后端微服务、Python 智能体服务和基础设施中间件四部分组成。

前端采用 React + TypeScript + Vite 技术栈，主要负责页面渲染、用户交互、参数录入、图表展示与 AI 对话界面构建。系统通过路由将功能模块划分为首页仪表盘、数据中心、计算分析、报告中心、AI 对话等页面，实现较清晰的前端功能结构。

Java 后端采用 Spring Cloud Alibaba 作为核心微服务框架，包括网关服务、认证服务、系统服务、数据服务和计算服务。其中，网关服务承担统一路由与跨域支持，认证服务负责用户登录与 Token 生成，数据服务负责基础数据管理与知识文档接入，计算服务则负责核心分析与结果记录。

Python 智能体服务采用 FastAPI 构建，主要承担知识检索、智能体工作流、动态报告生成与知识图谱辅助分析等功能。该服务与 Java 后端共同构成系统的智能分析能力。

基础设施层采用 MySQL、Redis、Nacos、Milvus 等组件。其中 MySQL 存储业务数据与历史记录，Redis 提供缓存与会话支撑，Nacos 负责服务注册发现与配置管理，Milvus 用于向量知识存储与检索。

整体架构兼顾了业务稳定性、模块扩展性和智能服务接入需求，能够较好支撑系统在专业计算与智能分析双重方向上的发展。

### 前端功能设计

前端部分围绕“数据录入、分析操作、结果展示、智能交互”四类任务展开设计。首页仪表盘页面用于展示项目数量、管道数量、泵站数量、油品数量以及多种图表结果；数据中心模块用于管理项目、管道、泵站、油品等基础信息；计算分析模块用于组织水力分析、泵站优化和敏感性分析的参数输入与结果展示；AI 模块则用于进行智能问答、知识录入与报告预览。

前端在交互设计上强调“参数可带入、结果可展示、过程可理解”。例如，计算页面支持直接从已有项目、管道、泵站和油品中带入参数，减少重复录入；结果页面通过描述面板、标签状态和 ECharts 图表展示分析结果，增强结果可读性；AI 对话页面支持多会话管理、模式切换、工具调用过程展示和人机协同交互，使复杂分析过程更加透明。

### 后端服务设计

后端服务采用职责分离的方式进行模块划分。

1. 网关服务 `pipeline-gateway` 作为统一入口，负责请求转发、跨域配置和服务发现。

2. 认证服务 `pipeline-auth` 负责用户登录、Token 生成与身份校验，并通过 Redis 实现分布式会话管理。

3. 数据服务 `pipeline-data` 负责项目、管道、泵站、油品、知识文档等业务对象的管理，是系统主要的数据支撑模块。

4. 计算服务 `pipeline-calculation` 负责水力分析、泵站优化、敏感性分析、计算历史归档和统计汇总，是系统专业能力的核心。

5. 系统服务 `pipeline-system` 用于用户、角色等基础系统管理。

通过这种分层方式，系统不同业务能力可以独立演进，也更适合后续的接口扩展与部署维护。

### 水力分析算法设计

水力分析模块核心由 `HydraulicAnalysisStrategy` 实现。该策略围绕管道水力学原理，按照工程计算逻辑依次完成参数校验、单位转换、管内流速计算、雷诺数计算、相对粗糙度计算、流态判断、摩阻系数计算、摩阻损失计算、水力坡降计算、总扬程计算、首站出站压头计算和末站进站压头计算等步骤。

该模块的关键特点是使用 BigDecimal 进行高精度处理，避免在专业工程计算中因浮点误差导致结果失真。系统通过对不同流态条件下的摩阻关系进行区分处理，使结果更符合工程分析需求。

### 泵站优化算法设计

泵站优化模块核心由 `OptimizationStrategy` 实现。系统预设多种 ZMI480 与 ZMI375 泵组组合方案，对不同方案逐一进行计算。每个方案均会计算总扬程、总压降、末站进站压头、方案可行性、总能耗与总成本等指标。

优化模块采用“在满足末站进站压头大于 0 的前提下，选择末站压力最小的可行方案”这一思路，从而减少能源浪费，提高方案节能性。系统最终返回推荐泵组组合及其说明，为运行调度提供参考依据。

### 敏感性分析算法设计

敏感性分析模块由 `SensitivityAnalysisStrategy` 实现。该策略以某一组基准工况参数为基础，针对指定变量按一定变化区间进行逐步扰动，并多次调用水力分析计算过程，获取不同变化比例下的分析结果。

系统根据不同结果之间的变化关系，计算变量对摩阻损失和末站进站压力的影响幅度，生成敏感系数、趋势判断和敏感性排名。该设计可以帮助用户识别关键影响变量，增强分析的解释性与针对性。

### 数据库设计

本系统数据库设计围绕“基础业务数据、计算历史数据、知识文档数据”三类内容展开。核心业务表包括：

1. `t_project`：用于存储项目信息，如项目编号、项目名称、负责人、创建时间等。
2. `t_pipeline`：用于存储管道参数，如长度、外径、壁厚、设计输量、起终点高程、粗糙度等。
3. `t_pump_station`：用于存储泵站参数，如泵效率、电机效率、排量、进站参数、不同泵型扬程等。
4. `t_oil_property`：用于存储油品物性数据，如名称、密度、黏度等。
5. `sys_user`：用于存储用户账号与认证相关信息。
6. `t_calculation_history`：用于存储水力分析、优化分析、AI 报告等历史结果。
7. `t_sensitivity_analysis`：用于存储敏感性分析相关结果。

通过上述表结构，系统能够同时支撑日常业务管理、专业计算过程与结果留存，也便于后续开展统计分析与报告生成。

### AI智能体工作流设计

系统智能体部分采用 FastAPI + LangGraph + LangChain 构建。其核心思路是：将用户输入交给大语言模型统一理解，由模型根据问题内容自主决定是否调用数据库查询工具、知识检索工具或计算工具，并在工具执行后综合输出最终结果。

在工具层面，系统实现了数据库查询工具、Java 服务调用工具、知识检索工具等，使智能体可以直接获取项目、管道、泵站、油品数据，可以调用后端水力分析与泵站优化接口，也可以从知识库中检索专业资料。

在知识检索层面，系统构建了包含文档处理、上下文分块、向量编码、Milvus 存储、混合检索、重排序、自适应检索决策的 RAG 管线，使问答结果具有更好的专业性与针对性。

### 动态报告生成设计

系统报告生成模块通过规则分析与大模型润色相结合的方式构建动态报告。报告生成服务会先从数据库和计算历史中提取项目数量、管道数量、泵站数量、历史统计结果等核心信息，再根据报告类型、分析对象、时间范围和输出风格生成结构化内容，包括摘要、关键发现、风险项、行动建议和结论等。

在需要时，系统可进一步调用大语言模型对报告语句进行润色，使报告内容在保持真实数据支撑的基础上具备更好的可读性和表达力。这种“规则优先、模型增强”的设计方式，可以降低纯生成式内容的不确定性，提高报告质量与可信度。

### 系统特色与实现效果

本系统在实现层面形成了以下几个较为鲜明的特点：

1. 行业场景明确。系统聚焦油气管道运行与能耗分析场景，功能设计具有较强的专业针对性。

3. 智能能力务实。AI 模块并非孤立聊天工具，而是与数据库、知识库和后端计算能力深度结合。

4. 工程可扩展性较强。系统采用微服务架构和独立智能体服务设计，后续可以继续扩展新算法、新模型和新业务场景。
